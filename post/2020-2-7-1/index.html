Paper Note - 2020/2/7

(PVLDB 2014) Large Scale Real-time Ridesharing with Service Guarantee on Road Networks.

Tong Yu
Feb 7, 2020 4 min read Paper Note - PVLDB

1. Scenario

Each request specifies two constraints :

  1. Waiting Time : defining the maximal time allowed between making the request and receiving the service.
  2. Service Constraint : defining the acceptable extra detour time from the shortest possible trip duration.

When a new request is received, it is evaluated immediately for server matching and scheduling.

2. Motivation

The goal is to schedule requests in real-time and minimize the servers’ travel times to complete all committed services while meeting service guarantees.

Real-time Ridesharing : Given a set of vehicles on the road network $G$ and a new incoming request $tr$ , find the vehicle that minimizes the overall trip cost for the augmented valid trip schedule.

3. Preliminaries

  1. A Road Network $G = <V,E,W>$ : consists of a vertex set $V$ and an edge set $E$. Each edge $(u,v)\in E$ is associated with a weight $W(u,v)$ indicating the traveling cost along the edge $(u,v)$ .
  2. A Path $p$ : is a vertex sequence.
  3. A Trip Request $tr = <s,e,w,\epsilon>$ :
    • $w$ : a maximal waiting time
    • $\epsilon$ : a service constraint, bounding the overall time from $s$ to $e$ by $(1+\epsilon)d(s,e)$
  4. A General Trip Schedule $S$ : trip set of size $m$ can be described in a sequence of $3m$ elements
    • $s_i$ : where a rider is picked up
    • $e_i$ : where a rider is dropped off
    • $r_i$ : where a request is received
  5. A Valid Trip Schedule : A valid trip schedule for a trip set $TR$ satisfies three conditions:
    1. Point order : $r_i \rightarrow s_i \rightarrow e_i$
    2. Waiting time constraint : $d_{T}(r_i,s_i) \leq w$
    3. Service constraint : $d_{T}(s_i,e_i) \leq (1+\epsilon)d(s_i, e_i)$
    4. Capacity limit
  6. A low maintenance cost indexing method is needed to filter out servers outside the waiting time constraint.

4. Method

4.1. Kinetic Tree Approach

4.1.1. Basic Tree Structure

4.1.2. Main Idea

When a server reaches a new pickup or dropoff location in the trip schedule, only those valid schedules which contain unfinished trips and share the same prefix so far (from the first pickup point of all the unfinished schedules to the current location in the trip schedule) need to be materialized. All other schedules become inactive and can be pruned from the tree.(Only a valid trip schedule can be extended to accommodate a new request.)

4.1.3. Challenge

The problem is how to determine:

  1. at which edge $s_k$ or $e_k$ can be inserted
  2. how to quickly prune the invalid trip schedules following that insertion

4.1.4. Inserting Pickup Location

  1. We can perform either a depth first search (DFS) or breadth first search (BFS) starting from the root node of the tree to generate all the candidate edge $(x_i, x_{i+1})$ to insert $s_k$ .
  2. At any given point, the “active” customers can be partitioned into two sets:
    • $S_1$ records those customers who need to be picked up
    • $S_2$ records the on-board customers who need to be dropped off
  3. For the sub-tree rooted at $x_i$ , the simple way is to first generate these two sets $S_1$ and $S_2$ . Then, when we insert $s_k$ , we need to ensure each condition associated with $S_1$ and $S_2$ are also satisfied.

4.2. Hotspot Based Optimization

Motivation : The main problem with the basic tree algorithm is the exponential explosion of the size of the tree when there are multiple pickup or dropoff locations close to each other.

  1. When we insert a pickup point $s_k$ to an edge $(x_i, x_{i+1})$ , we check if $d(x_{i+1}, s_k) \leq \theta$ for small $\theta$ . If so, $s_k$ is inserted into the node of $x_{i+1}$ .
  2. $s_k$ and $x_{i+1}$ are treated as one point called a hotspot in the tree, and an arbitrary schedule is chosen between the points in a hotspot.
  3. When a hotspot contains more than one point, a newly inserted point must be within $\theta$ to all points of the hotspot.
  4. A similar procedure can be followed for dropoff points and mixtures of pickup and dropoffs.
  5. Once a point is combined with any hotspot, we stop trying to insert it to any other edges.
